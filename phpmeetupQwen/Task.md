# Shipping Price Calculator — API Platform / Symfony

## User Story

As a backend service consumer, I want to calculate shipping prices for parcels within Croatia via a REST API, so that order/checkout systems can quote shipping costs based on destination ZIP, parcel weight and dimensions, and selected service tier (regular or priority).

## Acceptance Criteria

- A single POST endpoint exists under API Platform (e.g. `POST /api/shipping/quotes`) that accepts the following payload:

```json
{
  "postalCode": "10000",
  "weight": 2.4,
  "dimensions": { "length": 30, "width": 20, "height": 15 },
  "serviceType": "regular"
}
```

- Weight is in kilograms, dimensions in centimeters, `serviceType` is an enum: `regular` | `priority`.
- Response returns price, currency (`EUR`), service type, estimated delivery days, and a breakdown of how the price was composed (base, weight surcharge, dimensional surcharge, zone surcharge, priority multiplier):

```json
{
  "price": 5.49,
  "currency": "EUR",
  "serviceType": "regular",
  "estimatedDeliveryDays": 2,
  "breakdown": {
    "base": 3.50,
    "weightSurcharge": 0.99,
    "dimensionalSurcharge": 0.00,
    "zoneSurcharge": 1.00,
    "priorityMultiplier": 1.0
  }
}
```

- All Croatian postal codes (10000–53296 range, per Hrvatska Pošta) are accepted; unknown or invalid codes return `422` with a clear violation.
- Postal code → shipping zone mapping is configurable (data-driven, not hardcoded in PHP). Mainland, island, and remote/mountain zones can be priced differently.
- Pricing uses chargeable weight: `max(actualWeight, volumetricWeight)` where volumetric weight is `(L × W × H) / divisor` with the divisor configurable (typical 5000 or 6000).
- Priority shipping applies a configurable price multiplier and reduces estimated delivery days.
- Validation errors return RFC 7807 problem responses (API Platform default): negative/zero weight or dimensions, missing fields, oversized parcels exceeding configured per-service limits.
- OpenAPI/Swagger docs are auto-generated by API Platform and reflect the input DTO, output DTO, and all error codes.
- Endpoint is stateless — no DB writes for a quote. Reads only against zone/tariff configuration.

## Definition of Done

- DTOs are dedicated input/output classes (not Doctrine entities); business logic lives in a `ShippingPriceCalculator` service behind an interface.
- Unit tests cover the calculator (weight tiers, volumetric weight, zone resolution, priority multiplier, edge cases at tier boundaries).
- Functional/API tests cover the endpoint via API Platform's test client (happy path, invalid ZIP, oversized parcel, malformed payload, both service types).
- Static analysis passes at the project's configured level (PHPStan/Psalm).
- Code style passes (PHP-CS-Fixer / project standard).
- Tariff and zone data are seeded via fixtures or a config file; seeding is documented.
- README section added: endpoint contract, example requests, configuration knobs (volumetric divisor, zone definitions, tariff tables, priority multiplier).
- Code reviewed and merged to main.
- Deployed to staging; smoke-tested with at least one quote per zone and per service type.

## Additional Notes

- Currency is EUR — Croatia adopted the euro on Jan 1, 2023. Confirm with stakeholders whether prices are VAT-inclusive or VAT-exclusive before locking the contract; surface it explicitly in the response if needed.
- ZIP code dataset: Hrvatska Pošta publishes the official list. Decide whether to import it as a static dataset, a DB table, or a config file — DB table is recommended if zones will be edited via admin later.
- Island and remote-area surcharges are real for most Croatian carriers (Pag, Krk, Hvar, Korčula, Lastovo, Vis, mountain villages). Build the zone model to support this from day one even if all zones start with the same surcharge.
- Tariffs are usually tiered, not linear (e.g. 0–1 kg, 1–2 kg, 2–5 kg, 5–10 kg…). Model tariffs as ordered weight brackets, not as `pricePerKg`.
- Per-service hard limits are typical (e.g. priority refuses parcels above 30 kg or 150 cm longest side). Reject these with `422` rather than returning a quote.
- This is a calculator only — no carrier API integration, no label generation, no booking. If real-time carrier rates (Overseas, GLS, DPD, HP Express) are needed later, the `ShippingPriceCalculator` interface should be the seam: swap the implementation, keep the endpoint contract.
- Consider rate-limiting the endpoint if it's exposed to untrusted clients; quotes are cheap to compute but easy to abuse.
