# Shipping Price Calculator — API Platform / Symfony Architecture Plan

## 1. Project Overview

A stateless REST API for calculating shipping prices for parcels within Croatia, built on Symfony + API Platform. The service accepts parcel details (destination ZIP, weight, dimensions, service tier) and returns a quoted price with a detailed breakdown. All pricing configuration (zones, tariffs, divisors, multipliers) is data-driven and externalized from PHP code.

## 6. Service Architecture

### 6.1 Calculator Interface

```php
// src/Service/ShippingPriceCalculatorInterface.php
namespace App\Service;

use App\DTO\ShippingQuoteRequest;
use App\DTO\ShippingQuoteResponse;

interface ShippingPriceCalculatorInterface
{
    public function calculate(ShippingQuoteRequest $request): ShippingQuoteResponse;
}
```

### 6.2 Calculator Implementation Logic (Pseudocode)

```
calculate(request):
    1. Validate request against limits (max weight, dimensions) → throw if exceeded
    2. Resolve zone from postalCode → get surcharge + delivery_days_base
    3. Load tariff table for serviceType
    4. Compute volumetricWeight = (L × W × H) / volumetric_divisor
    5. chargeableWeight = max(actualWeight, volumetricWeight)
    6. Find weight tier: find first tier where min ≤ chargeableWeight ≤ max
    7. basePrice = tier.base_price
    8. price = (basePrice + zoneSurcharge) * priorityMultiplier
       where priorityMultiplier = (request.priority ? settings.default_priority_multiplier : 1.0)
    9. estimatedDeliveryDays = zone.delivery_days_base - (request.priority ? 1 : 0)
    10. Return ShippingQuoteResponse with breakdown
```

### 6.3 Pricing Breakdown Calculation Detail

| Component | Formula |
|-----------|---------|
| `base` | Tariff tier base_price for chargeable weight bracket |
| `weightSurcharge` | (tierPrice - previousTierBase) if in a higher bracket, else 0. Alternatively: `(chargeableWeight - baseWeightThreshold) * marginalRate` |
| `dimensionalSurcharge` | max(0, volumetricWeight - actualWeight) × perKgDimensionalRate (default 0.00 unless configured) |
| `zoneSurcharge` | Zone-specific flat surcharge from zones.yaml |
| `priorityMultiplier` | 1.0 for regular, settings.default_priority_multiplier for priority |

```php
{
    // ... fields as defined in section 4.1
```

### 7.2 Processor (No DB Write)

API Platform's processor chain is configured to delegate to the calculator:

```php
// src/Processor/ShippingQuoteProcessor.php
namespace App\Processor;

use ApiPlatform\Metadata\IriConverterInterface;
use ApiPlatform\State\ProcessorInterface;
use App\DTO\ShippingQuoteRequest;
use App\DTO\ShippingQuoteResponse;
use App\Service\ShippingPriceCalculatorInterface;

class ShippingQuoteProcessor implements ProcessorInterface
{
    public function __construct(private ShippingPriceCalculatorInterface $calculator) {}

    public function process(mixed $data): ShippingQuoteResponse
    {
        assert($data instanceof ShippingQuoteRequest);
        return $this->calculator->calculate($data);
    }
}
```

### 7.3 Normalizer (Output DTO → JSON)

```php
// src/Serializer/ShippingQuoteNormalizer.php
class ShippingQuoteNormalizer implements NormalizerInterface
{
    public function normalize(mixed $object, ?string $format = null, array $context = []): array
    {
        assert($object instanceof ShippingQuoteResponse);
        return [
            'price' => $object->price,
            'currency' => $object->currency,
            'serviceType' => $object->serviceType,
            'estimatedDeliveryDays' => $object->estimatedDeliveryDays,
            'breakdown' => [
                'base' => $object->breakdown->base,
                'weightSurcharge' => $object->breakdown->weightSurcharge,
                'dimensionalSurcharge' => $object->breakdown->dimensionalSurcharge,
                'zoneSurcharge' => $object->breakdown->zoneSurcharge,
                'priorityMultiplier' => $object->breakdown->priorityMultiplier,
            ],
        ];
    }

    public function supportsNormalization(mixed $data, ?string $format = null): bool
    {
        return $data instanceof ShippingQuoteResponse;
    }
}
```

## 8. Validation Strategy

### 8.1 Input Validation (Symfony Validator → 422 via API Platform)

| Field | Constraint | Error on |
|-------|-----------|----------|
| `postalCode` | NotBlank + 5-digit regex | Missing or wrong format |
| `weight` | Positive (>0) | ≤ 0 or null |
| `dimensions.length` | Positive (>0) | ≤ 0 or null |
| `dimensions.width` | Positive (>0) | ≤ 0 or null |
| `dimensions.height` | Positive (>0) | ≤ 0 or null |
| `serviceType` | Enum['regular','priority'] | Other value |

### 8.2 Business Validation (Calculator → 422)

| Rule | Exception / Response |
|------|---------------------|
| Postal code not in any zone | `UnknownPostalCodeException` → 422 with "Unknown postal code" |
| Weight exceeds max for service type | `OversizedParcelException` → 422 with "Parcel exceeds maximum weight" |
| Longest side exceeds limit | `OversizedParcelException` → 422 |
| Sum of dimensions exceeds limit | `OversizedParcelException` → 422 |

Validation order in calculator: validate dimensions limits → validate weight limits → proceed with calculation.

### 8.3 RFC 7807 Problem Detail Response (Example)

```json
{
    "@type": "Problem",
    "title": "Unprocessable Entity",
    "status": 422,
    "detail": "postalCode: This postal code is not recognized by Hrvatska Pošta.",
    "violations": [
        "postalCode"
    ]
}
```

## 10. OpenAPI/Swagger Documentation

API Platform auto-generates OpenAPI 3.0 specs at `/api/docs.json`. The endpoint is documented with:

- **Path**: `POST /api/shipping/quotes`
- **Request body schema**: ShippingQuoteRequest DTO with all constraints
- **Response 200**: ShippingQuoteResponse schema
- **Response 422**: Problem detail with violations array
- **Response 400**: Bad request (malformed JSON)

The `#[OpenApiOperation]` attribute on the input DTO configures summary, description, and example values.

## 11. Configuration Knobs (README Section)

All configuration is externalized to `config/shipping/`:

| Knob | Config File | Description |
|------|-------------|-------------|
| Volumetric divisor | `settings.json` → `volumetric_divisor` | Divides L×W×H to get volumetric weight (5000 or 6000) |
| Zone definitions | `zones.yaml` | Postal code ranges mapped to zones with surcharge and base delivery days |
| Tariff tables | `tariffs_{serviceType}.json` | Ordered weight brackets with base prices |
| Priority multiplier | `settings.json` → `default_priority_multiplier` | Multiplier applied to price for priority service (e.g., 1.5) |
| Service limits | `limits.json` | Max weight, max longest side, max dimension sum per service type |
| Currency | `settings.json` → `currency` | Default EUR; change if serving non-Euro zones |
| VAT flag | `settings.json` → `vat_inclusive` | Whether displayed prices include VAT |

**To update tariffs**: Edit the JSON files and restart the PHP worker (or rely on file watcher in dev mode). No code deploy needed.

## 14. Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Config storage | YAML + JSON files (not DB) | Stateless calculator; no write needed; simpler deployment |
| No Doctrine entities | Pure DTOs | Requirement: calculation is stateless, no persistence |
| Chargeable weight = max(actual, volumetric) | Industry standard | Matches carrier practices (DHL, FedEx use similar) |
| Tiered tariffs (not per-KG) | Ordered brackets | Matches real carrier pricing; non-linear |
| Zone mapping externalized | YAML config | Supports mainland/island/remote without code changes |
| Interface-based calculator | `ShippingPriceCalculatorInterface` | Swap implementation later (e.g., real carrier API) |
| RFC 7807 errors | API Platform default | Standard problem details format; consumed by any client |
| Currency EUR | Hardcoded default | Croatia adopted euro in 2023; configurable if needed |

## 15. Future Extensibility

- **Carrier rate integration**: Replace `ShippingPriceCalculator` implementation with a `CarrierRateCalculator` that calls external APIs while keeping the same interface and endpoint contract.
- **Admin zone management**: Swap file-based zone config to Doctrine-backed `Zone` entity with admin CRUD endpoints if non-technical staff need to update zones.
- **VAT handling**: Add `vatPercent` and `priceIncludingVat` fields to output DTO if VAT disclosure is required.
- **Bulk quotes**: Add `POST /api/shipping/bulk-quotes` accepting an array of requests, reusing the same calculator service.
- **Caching**: Add APCu/file cache to repositories for zone/tariff config to avoid file reads on every request in high-throughput scenarios.
